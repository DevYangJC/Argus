# Argus 项目文档 V1.0

> Argus 是一个 **RAG 知识库平台**的教学项目，从零搭建，逐步迭代。
> V1.0 完成了项目骨架、用户认证与用户管理模块。

---

## 一、项目概述

### 1.1 项目定位

Argus 意为"百眼巨人"，寓意平台能洞察所有文档内容。最终形态是一个支持文档上传、向量检索、大模型对话的 RAG（Retrieval-Augmented Generation）知识库平台。

V1.0 作为起点，聚焦于**后端基础设施搭建**和**用户体系**，为后续业务模块提供稳固的底座。

### 1.2 V1.0 功能清单

| 模块 | 功能 | 状态 |
|------|------|------|
| 基础设施 | 统一响应体、全局异常处理、枚举规范 | 完成 |
| 基础设施 | Knife4j 接口文档 | 完成 |
| 认证模块 | 注册 / 登录 / 登出 | 完成 |
| 认证模块 | JWT access token + Refresh Token 双令牌 | 完成 |
| 认证模块 | 令牌刷新 | 完成 |

| 用户模块 | 管理员-用户列表/详情 | 完成 |

### 4.2 users 表字段说明

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键 |
| user_code | VARCHAR(64) | NOT NULL | 用户编码，注册时同 username |
| username | VARCHAR(64) | NOT NULL, UNIQUE | 登录用户名 |
| email | VARCHAR(128) | NOT NULL, UNIQUE | 邮箱 |
| display_name | VARCHAR(128) | NOT NULL | 显示名称 |
| password_hash | VARCHAR(256) | NOT NULL | BCrypt 哈希 |
| system_role | VARCHAR(16) | NOT NULL, DEFAULT 'USER' | 系统角色：ADMIN / USER |
| status | VARCHAR(16) | NOT NULL, DEFAULT 'ACTIVE' | 账号状态：ACTIVE / DISABLED |
| must_change_password | BOOLEAN | NOT NULL, DEFAULT FALSE | 是否下次登录强制改密码 |
| last_login_at | TIMESTAMP | - | 最后登录时间 |
| created_at | TIMESTAMP | NOT NULL, DEFAULT now() | 创建时间 |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT now() | 更新时间 |

### 4.3 user_refresh_tokens 表字段说明

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键 |
| user_id | BIGINT | NOT NULL, FK | 关联 users.id |
| token_id | VARCHAR(64) | NOT NULL, INDEX | UUID 去横线，明文索引 |
| token_hash | VARCHAR(256) | NOT NULL | BCrypt 哈希后的 token |
| expires_at | TIMESTAMP | NOT NULL | 过期时间 |
| revoked_at | TIMESTAMP | - | 吊销时间，NULL = 有效 |
| created_at | TIMESTAMP | NOT NULL, DEFAULT now() | 创建时间 |

**索引设计**：
- `uq_users_username` / `uq_users_email` — 唯一约束索引，用于登录查找
- `idx_refresh_token_token_id` — 单列索引，按 token_id 快速查找
- `idx_refresh_token_user_active` — 复合索引 `(user_id, revoked_at, expires_at)`，用于吊销和统计

## 五、核心设计决策

### 5.1 不使用 JdbcTemplate，统一使用 MyBatis-Plus

V1.0 最初用 `JdbcTemplate` 手写 SQL，后期全部改造为 MyBatis-Plus：

| 对比维度 | JdbcTemplate | MyBatis-Plus |
|---------|-------------|--------------|
| 查询方式 | 手写 SQL 字符串 | `LambdaQueryWrapper` 类型安全 |
| 结果映射 | RowMapper 手动映射 | 实体自动映射 |
| 插入回填 ID | `RETURNING id`（PG 特有） | `insert` 自动回填 |
| 复杂 SQL | 内嵌字符串 | XML 文件分离 |
| 枚举处理 | `status.name()` 手动 | `EnumTypeHandler` 自动 |

### 5.2 双令牌认证机制

```
access token (JWT)          refresh token (opaque)
├── 有效期 30 分钟           ├── 有效期 14 天
├── Bearer header 传输       ├── httpOnly Cookie 传输
├── 无状态，不查库           ├── 数据库存储 + BCrypt 哈希
└── 过期后用 refresh 换      └── 吊销实现服务端登出控制
```

- **access token**：短期，存于内存，过期后前端用 refresh 接口无感刷新
- **refresh token**：长期，存于 `user_refresh_tokens` 表，登录时吊销旧 token（单设备登录）

### 5.3 全局异常处理

```java
// 业务代码直接抛出自定义异常
throw new BusinessException("用户不存在");          // → 400
throw new UnauthorizedException("未登录");         // → 401
throw new ForbiddenException("非管理员");           // → 403

// GlobalExceptionHandler 统一转换为 ApiResponse
@ExceptionHandler(BusinessException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)
public ApiResponse<Void> handleBusinessException(...) { ... }
```

### 5.4 统一响应体 (record)

```java
// 所有控制器统一返回格式
public record ApiResponse<T>(boolean success, T data, String message) {}

// 成功
ApiResponse.success(userList)
// → {"success": true, "data": [...], "message": null}

// 失败
ApiResponse<>(false, null, "用户不存在")
// → {"success": false, "data": null, "message": "用户不存在"}
```

### 5.5 枚举数据库映射

所有枚举在数据库中存储为 `VARCHAR` 类型的 `.name()` 值（如 `"ADMIN"`、`"ACTIVE"`），MyBatis-Plus 通过 `EnumTypeHandler` 自动转换：

```yaml
mybatis-plus:
  configuration:
    default-enum-type-handler: org.apache.ibatis.type.EnumTypeHandler
```

## 六、API 接口文档

### 6.1 认证模块 - `/api/auth`

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| POST | `/api/auth/register` | 否 | 注册新用户 |
| POST | `/api/auth/login` | 否 | 登录，返回 accessToken + Cookie 下发 refreshToken |
| POST | `/api/auth/refresh` | 否 | 从 Cookie 读 refreshToken，刷新双令牌 |
| POST | `/api/auth/logout` | 否 | 吊销 refreshToken，清除 Cookie |
| GET | `/api/auth/me` | 是 | 获取当前用户信息 |

**登录接口示例**：
```
POST /api/auth/login
Content-Type: application/json

{"loginId": "admin", "password": "Admin@123456"}

Response 200:
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOi...",
    "currentUser": {
      "userId": 1,
      "userCode": "admin",
      "displayName": "管理员",
      "systemRole": "ADMIN",
      "mustChangePassword": false
    }
  }
}
```

### 6.2 账户模块 - `/api/account`

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| POST | `/api/account/change-password` | 是 | 修改当前用户密码 |

### 6.3 管理员模块 - `/api/admin/users`

| 方法 | 路径 | 认证 | 权限 | 说明 |
|------|------|------|------|------|
| GET | `/api/admin/users` | 是 | ADMIN | 用户列表 |
| GET | `/api/admin/users/{userId}` | 是 | ADMIN | 用户详情 |
| PATCH | `/api/admin/users/{userId}/status` | 是 | ADMIN | 启用/禁用用户 |

## 七、认证流程详解

### 7.1 登录流程

```
Client                          Server
  │                               │
  │  POST /api/auth/login         │
  │  {loginId, password}          │
  │ ─────────────────────────────>│
  │                               │  1. 校验账号密码
  │                               │  2. 吊销旧 refresh token（单设备登录）
  │                               │  3. 签发 accessToken（JWT, 30min）
  │                               │  4. 签发 refreshToken（opaque, 14天）
  │                               │  5. refreshToken 存入 DB（BCrypt 哈希）
  │                               │  6. update last_login_at
  │  accessToken + Set-Cookie    │
  │ <─────────────────────────────│
```

### 7.2 请求认证流程

```
Client                          Server
  │                               │
  │  GET /api/admin/users          │
  │  Authorization: Bearer xxx    │
  │ ─────────────────────────────>│
  │                               │  JwtAuthenticationFilter
  │                               │  1. 提取 Bearer token
  │                               │  2. 解析 JWT → AuthenticatedUser
  │                               │  3. 设置 request attribute
  │                               │
  │                               │  AdminUserController
  │                               │  4. requireSystemAdmin(request)
  │                               │  5. 校验 systemRole == ADMIN
  │                               │  6. 执行业务逻辑
  │  ApiResponse<List<...>>      │
  │ <─────────────────────────────│
```

### 7.3 令牌刷新流程

```
Client（前端拦截器, 无感）       Server
  │                               │
  │  accessToken 过期 (401)
  │  POST /api/auth/refresh       │
  │  Cookie: refreshToken         │
  │ ─────────────────────────────>│
  │                               │  1. 从 DB 查找 refreshToken
  │                               │  2. BCrypt 比对
  │                               │  3. 检查未吊销 + 未过期
  │                               │  4. 吊销旧 token（rotation）
  │                               │  5. 签发新 accessToken + refreshToken
  │  新 accessToken + Cookie     │
  │ <─────────────────────────────│
  │                               │
  │  用新 accessToken 重试请求    │
```
